# main 브랜치 코드리뷰

## 먼저 총평

현재 코드는 "Todo 앱의 시작 화면" 정도까지는 만들어졌지만, 아직 실제 Todo 앱이라고 부르기에는 기능 구현이 거의 없는 상태입니다.  
초보자가 시작할 때 자주 나오는 형태이고, 지금 시점에서 가장 중요한 것은 "조금 더 예쁘게 만들기"보다 "역할을 나누고, API와 DOM을 연결하는 구조를 잡는 것"입니다.

좋은 점도 있습니다.

- Tailwind CSS를 이미 연결해 두었습니다.
- 입력창과 버튼이 있어서 Todo 입력 UI의 방향은 잡혀 있습니다.
- `main.js`와 `style.css`를 분리해 두어서 확장할 기반은 있습니다.

하지만 현재 상태에서는 다음 문제가 큽니다.

- 화면 구조는 있는데 실제 기능이 없습니다.
- HTML 안에 앞으로 JS가 맡아야 할 책임까지 섞일 가능성이 큽니다.
- 재사용 가능한 구조가 없어서 기능이 늘어나면 코드가 빠르게 복잡해질 수 있습니다.
- Tailwind를 쓰고 있지만, 어떤 부분을 HTML에서 해결하고 어떤 부분을 CSS에서 해결할지 기준이 아직 약합니다.

아래는 파일별 리뷰입니다.

---

## 주요 리뷰 포인트

### 1. `src/main.js:1`

현재 JS 파일에는 아래 한 줄만 있습니다.

```js
import "./style.css";
```

이 상태에서는 Todo 추가, 목록 조회, 완료 토글, 삭제, 수정 같은 핵심 기능이 하나도 동작하지 않습니다.

왜 중요한가?

- Todo 앱의 핵심은 "데이터를 가져오고, 화면에 그리고, 사용자 이벤트를 처리하는 것"입니다.
- 그런데 지금은 JS가 사실상 비어 있으므로 앱의 핵심 책임이 아직 시작되지 않은 상태입니다.

개선 방향

- 최소한 아래 역할은 JS에서 분리해서 가져가야 합니다.
- API 호출
- DOM 선택
- 화면 렌더링
- 이벤트 연결

---

### 2. `index.html:10-30`

`#app` 안에 제목, 폼, `template`이 들어 있습니다. 시작점으로는 괜찮지만 구조가 아직 Todo 앱 목적에 비해 부족합니다.

현재 아쉬운 점

- Todo 목록이 실제로 그려질 컨테이너가 명확하지 않습니다.
- `template` 태그가 있지만 `id`도 없고, 실제 JS에서 사용할 수 있는 구조가 아닙니다.
- 필터, 빈 목록 메시지, 로딩/에러 메시지 영역이 없습니다.

왜 중요한가?

- 화면에 필요한 영역이 먼저 명확해야 JS도 덜 복잡해집니다.
- "어디에 렌더링할지"가 분명하지 않으면 `main.js`에서 DOM 조작 코드가 금방 지저분해집니다.

개선 방향

- 목록 렌더링용 영역을 명확히 둡니다.
- 예: `#todo-list`, `#empty-message`, `#error-message`
- `template`를 쓸 거라면 `id="todo-item-template"`처럼 이름을 붙입니다.

---

### 3. `index.html:12-15`

`form`에 `action=""`이 들어 있습니다.

왜 아쉬운가?

- 현재 앱은 서버가 HTML 페이지를 다시 내려주는 방식이 아니라, JS로 동작하는 웹앱입니다.
- 이런 경우 `form` 제출은 기본 동작을 막고 `fetch`로 처리하는 것이 일반적입니다.
- `action=""`은 초보자에게 "어디론가 제출하는 폼"처럼 오해를 줄 수 있습니다.

개선 방향

- `action`은 빼고, `submit` 이벤트에서 `event.preventDefault()`를 사용합니다.

---

### 4. `index.html:16-24`

입력창과 버튼은 있지만 접근성과 확장성을 조금 더 고려할 필요가 있습니다.

아쉬운 점

- 입력창에 연결된 `<label>`이 없습니다.
- 지금은 제목만 입력하지만, 실제 API에는 설명, 우선순위, 마감일도 있습니다.
- 나중에 수정 기능까지 붙이면 폼 구조를 다시 크게 바꿔야 합니다.

왜 중요한가?

- `placeholder`는 안내 문구이지, 라벨을 완전히 대신하지 못합니다.
- 처음부터 "폼은 하나의 역할을 가진 UI"로 설계하면 수정할 때 덜 힘듭니다.

개선 방향

- `label`을 추가합니다.
- Todo 생성 폼에 필요한 필드를 처음부터 분리해 둡니다.
- 예: 제목, 설명, 우선순위, 마감일

---

### 5. `index.html:26-30`

`template` 안에 `ul > li` 구조가 들어 있습니다.

이 구조는 추천하지 않습니다.

이유

- 보통 목록 전체를 감싸는 `ul`은 화면에 한 번만 있어야 합니다.
- 반복되는 것은 `li`입니다.
- 현재처럼 `template` 안에 `ul`이 있으면 Todo 하나를 만들 때마다 `ul`까지 반복 생성될 가능성이 큽니다.

개선 방향

- 실제 화면에는 `ul`을 하나 두고,
- `template` 안에는 `li` 하나만 두는 편이 좋습니다.

좋은 예시 구조

```html
<ul id="todo-list"></ul>

<template id="todo-item-template">
  <li class="...">
    ...
  </li>
</template>
```

### 6. `src/style.css:3-7`

커스텀 테마 변수를 일부 정의했지만 아직 사용 기준이 뚜렷하지 않습니다.

아쉬운 점

- `--color-point-purple`는 현재 사용되지 않습니다.
- 색 이름과 실제 의도가 섞여 있습니다.
- 초보 단계에서는 "왜 변수를 만들었는지"가 분명해야 유지보수가 쉬워집니다.

왜 중요한가?

- 변수를 만들었는데 실제로 안 쓰면 코드만 늘고 의미는 약해집니다.
- 특히 Tailwind 중심 프로젝트에서는 CSS 변수를 무분별하게 늘리지 않는 것이 좋습니다.

개선 방향

- 정말 반복해서 쓸 값만 남깁니다.
- Tailwind 유틸리티로 충분한 부분은 HTML 클래스에서 해결합니다.
- CSS는 `body` 기본 레이아웃, 정말 공통인 스타일 정도만 남깁니다.

---

### 7. `src/style.css:9-13`

현재 CSS는 `body` 레이아웃만 담당하고 있습니다. 이 방향 자체는 괜찮습니다.

좋은 점

- 모든 스타일을 CSS에 몰아넣지 않았습니다.
- Tailwind를 중심으로 가려는 방향이 보입니다.

보완할 점

- 이 원칙을 끝까지 유지해야 합니다.
- 컴포넌트 하나를 만들 때마다 CSS 파일에 클래스를 추가하기 시작하면 Tailwind를 쓰는 장점이 줄어듭니다.

추천 원칙

- 가능한 한 HTML에서 Tailwind 클래스 사용
- CSS는 최소한만 사용
- 반복되는 UI 패턴이 많아질 때만 CSS 또는 컴포넌트 단위 추상화 고민

### 8. 전체 구조 관점: 응집도와 결합도

초보자에게 가장 중요한 리뷰 포인트는 여기입니다.

#### 현재 상태의 위험

지금 기능이 거의 없기 때문에 아직 문제는 크게 드러나지 않았습니다.  
하지만 이 상태에서 바로 `main.js`에 모든 코드를 한 파일로 몰아 쓰기 시작하면 아래 문제가 생깁니다.

- API 주소 문자열이 여기저기 흩어짐
- `fetch` 코드가 여러 군데 중복됨
- `document.querySelector(...)`가 파일 전체에 퍼짐
- 렌더링 코드와 이벤트 코드가 섞임
- 수정 기능이 들어갈 때 기존 코드가 쉽게 망가짐

#### 응집도를 높인다는 뜻

서로 관련 있는 코드를 가까운 곳에 모으는 것입니다.

예시

- API 호출 코드는 `todoApi.js`
- Todo 목록 렌더링은 `renderTodos.js`
- DOM 요소 선택은 `elements.js`

#### 결합도를 낮춘다는 뜻

한 파일이 다른 파일의 내부 구조를 너무 많이 알지 않게 만드는 것입니다.

예시

- `main.js`는 "Todo 목록 다시 그려줘"만 알면 되고,
- `renderTodos.js` 내부에서 어떤 HTML을 만드는지는 몰라도 됩니다.

이 구조가 되면

- 수정이 쉬워지고
- 버그가 줄고
- 다른 사람이 읽기도 쉬워집니다.

3. `template` 구조를 `li` 단위로 바꾸세요.

5. Tailwind 중심으로 UI를 만들고 CSS는 최소화하세요.

2. 에러 처리 문구 추가
3. 수정 기능용 폼 또는 모달 구조 고민
4. 빈 목록 상태 메시지 추가

## 마무리 평가

현재 코드는 "나쁜 코드"라기보다 "아직 시작 단계의 코드"에 가깝습니다.  
문제는 실력이 부족해서라기보다, 아직 앱의 책임 분리가 시작되지 않았다는 점입니다.

지금 단계에서 가장 중요한 목표는 아래입니다.

- HTML은 화면의 뼈대
- Tailwind는 대부분의 스타일
- CSS는 최소한의 공통 스타일
- JS는 API, 이벤트, 렌더링을 역할별로 분리

이 기준만 잘 잡아도 이후 구현 난이도가 크게 내려갑니다.
